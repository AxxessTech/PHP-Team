# Feature Branch Workflow

The core idea behind the Feature Branch Workflow is that all feature development should take place in a dedicated branch instead of the master branch. This encapsulation makes it easy for multiple developers to work on a particular feature without disturbing the main codebase. It also means the master branch will never contain broken code, which is a huge advantage for continuous integration environments.

Encapsulating feature development also makes it possible to leverage pull requests, which are a way to initiate discussions around a branch. They give other developers the opportunity to sign off on a feature before it gets integrated into the official project. Or, if you get stuck in the middle of a feature, you can open a pull request asking for suggestions from your colleagues. The point is, pull requests make it incredibly easy for your team to comment on each others work.


# Starting a new feature

It is prefered to create a new feature branch directly from a Jira ticket so that features can be tracked back to the original requirement.

# i) Creating a new feature branch from a Jira ticket:

from JIRA > "Create Branch":
    Repository: CAHPS/admin3
    Branch Type: [Bugfix|Feature]
    Branch from: Master
    Branch name: PE-123-my-feature-branch

# Then, from the CLI:

$ git fetch origin feature/PE-123-my-feature-branch:feature/PE-123-my-feature-branch
$ git checkout feature/PE-123-my-feature-branch

# ii) Creating a new feature branch from the CLI:

$ git checkout master
$ git pull origin master

$ git checkout -b feature/my-feature-branch

$ git push origin feature/my-feature-branch

# Working on the feature branch

On this branch, edit, stage, and commit changes in the usual fashion, building up the feature with as many commits as necessary.

$ git add <some-file>
$ git add <some-file>
$ git commit -m "Started work on feature"

$ git add <some-file>
$ git commit -m "Did some more work on feature"

TIP

It's a good idea to periodically push the feature branch up to the central repository. This serves as a convenient backup, when collaborating with other developers, this would give them access to view commits to the new branch.

$ git push origin feature/my-feature-branch

Try to do this at least once a week.


Work on the feature and make commits like you would any time you use Git.


TIP

If the development of the feature branch spans a significant period of time, you should periodically merge the master branch into the feature branch and resolve any merge conflicts.

$ git checkout feature/my-feature-branch
$ git merge master

Resolve any merge conflics.


# Completing the feature

Run the test suite.

$ php artisan test

Merge the master branch into the feature branch and resolve any merge conflicts.

$ git checkout feature/my-feature-branch
$ git merge master

Resolve any merge conflics.


Next, if any fontend assets were changed during the development of your feature, they need to be compiled.

# Run all Mix tasks and minify output...
$ npm run production

$ git add .
$ git commit -m "Compiled production assets"

Once complete, make one final push to the central repository.

$ git push origin feature/my-feature-branch

Lastly, if there are multiple commits in your feature branch, you should consider squashing your commits. This will enforce a cohesive Git history of feature merges.

$ git rebase --interactive [SHA]

Note: That [SHA] is the hash of the commit just before the first one you want to rewrite from.

# Interactive rebasing session
pick 1230001 Started work on feature
squash 1230002 Did some more work on feature

# Making a Pull Request

Push your completed feature branch to the central repository. Note: Use the --force flag if you squashed any commits on your feature branch.

$ git push --force origin feature/my-feature-branch

from Stash > "Create pull request":
    Source: feature/my-feature-branch
    Destination: master
    Reviewers: [A colleague]

# Final steps

If you have been assigned to review a PR, you must be satisfied with the changed code before approving and merging. Ask your self, has the spirit of the ticket been satisfied, is the code written cleanly, have any bugs been introduced? Depending on the size of the change, you can choose to simply visually inspect the code in Stash or for larger changes, pull down the feature branch and test on your local system.

# Pull down the feature branch
$ git fetch origin feature/my-feature-branch:feature/PE-123-my-feature-branch
$ git checkout feature/my-feature-branch

# Approve and Merge

Once satisfied you can approve the PR. Then ensure that "Delete source branch after merging" is checked before hitting the "Merge" button.


# Clean up locally

$ git checkout master
$ git pull origin master
$ git branch -d feature/my-feature-branch

Note that Git should not complain about the git branch -d, since new feature is now accessible from the master branch.
